### Debug Log References

**🚨 Production bug discovered during Task 4 (smoke testing) — pre-existing latent defect from Stories 1.4–1.6:**

- **Symptom (caught by owner on first page load):**
  ```
  ModuleNotFoundError: No module named 'app'
  File "/app/app/main.py", line 20, in <module>
      from app.duration import format_duration
  ```
- **Root cause:** Dockerfile has `WORKDIR /app` and copies `app/` to `/app/app/`. The CMD `streamlit run app/main.py` causes Streamlit to add `/app/app/` (script's directory) to `sys.path` — but **NOT `/app/`**. The app uses absolute imports (`from app.duration import …`), so Python cannot resolve the `app` package, and the script crashes on first user request.
- **Why this wasn't caught earlier:**
  1. **pytest** runs from the project root with the root on `sys.path` → imports work locally. Tests pass 85/85.
  2. **Streamlit's `/_stcore/health` endpoint** responds at the framework level **before** the user script is loaded → `curl /_stcore/health` returns `200 OK` even when the script crashes.
  3. **Railway's `SUCCESS` deploy state** only confirms `pip install` + service startup — not script-level execution.
  4. The bug was latent in **every deploy from Story 1.4 onward** (when the first `from app.X` import was added to main.py). It surfaced only when a user actually opened the page.
- **Fix:** added `PYTHONPATH=/app` to Dockerfile ENV directive (commit `58767c8`). Makes `app/` importable in any execution context, regardless of how Streamlit arranges `sys.path`.
- **Validation:** local pytest 85/85 still passes (no regression — pytest doesn't use the Dockerfile env). Live verification pending @devops push + Railway redeploy.
