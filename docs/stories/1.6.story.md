## Story

**As** an end user,
**I want** to receive clear, friendly PT-BR error messages whenever something goes wrong,
**so that** I understand what happened and what to try next, without seeing technical jargon or stack traces.

## Acceptance Criteria

> Source: `docs/prd.md` Epic 1, Story 1.6 (verbatim).

1. **AC1:** A centralized error-message catalog (`app/errors.py` or equivalent) is created mapping internal error categories to user-facing PT-BR messages. Categories at minimum: **`INVALID_URL`, `UNSUPPORTED_SOURCE`, `VIDEO_TOO_LONG`, `DOWNLOAD_FAILED`, `TRANSCRIPTION_FAILED`, `TIMEOUT`, `UNEXPECTED`**.
2. **AC2:** All exception-raising paths in the application catch their domain-specific exceptions and map them to one of the above categories before surfacing to the UI.
3. **AC3:** User-facing error messages contain **NO Python stack trace, NO HTTP status codes, NO library names, NO file paths, NO API endpoint URLs**.
4. **AC4:** Server-side logs (stdout) include the full technical detail (exception type, traceback, library context) for debugging — but **NOT the source URL or any transcript content**.
5. **AC5:** An `UNEXPECTED` catch-all wraps the top-level Streamlit handler so any uncaught exception produces a graceful PT-BR message (**"Algo deu errado. Tente novamente em alguns minutos."**) rather than a Streamlit error stack-trace dump.
6. **AC6:** After any error is displayed, the input field remains usable and the user can submit a new URL without page reload.
7. **AC7:** Manual test verification: submit a malformed URL, an unreachable URL, a video over 60 min, and (if possible) an invalid OpenAI API key configuration — all four cases produce friendly PT-BR errors with no technical leakage.

### Architecture Anchor — Top-Level Catch-All (architecture §6.3)

The architect's pseudo-code in §6.3:

```python
# main.py — outermost wrapper around the pipeline
try:
    run_pipeline(url)
except TranscricaoError as e:
    st.error(e.user_message_pt)
except Exception as e:
    logger.exception("Unexpected error in pipeline")
    st.error(UnexpectedError().user_message_pt)
```

Story 1.6's implementation differs slightly: instead of refactoring the inline pipeline into a `run_pipeline(url)` function, we wrap the existing `if submitted:` block. This is functionally equivalent and avoids unnecessary churn. The `except TranscricaoError: pass` is defensive (the inner stage-handlers already handle these via `st.error + st.stop`); the outer `except Exception:` is the new safety net.

  - [Source: `docs/architecture.md` §6.3 — top-level catch-all]

### TIMEOUT Category — Resolution

PRD AC1 says category list at minimum includes `TIMEOUT`. Architecture §6.1 says:

> "TIMEOUT is mentioned in Story 1.6 AC1. At MVP the synchronous flow has no explicit timeout layer (Streamlit holds the WebSocket; Whisper has its own SDK-level timeout). If a wall-clock cap is added in Phase 2, a `TimeoutError(TranscricaoError)` subclass will be added then. For now, timeouts surface as `TRANSCRIPTION_FAILED` or `DOWNLOAD_FAILED` depending on which stage stalls — covered."

**Resolution:** add the `TimeoutError` class to errors.py for **catalog completeness** (matches PRD AC1's enumerated list), but do NOT actively raise it anywhere. Document the "reserved" status in the docstring. Phase 2's wall-clock timeout layer can start raising it without a schema migration.
