## Story

**As** an end user,
**I want** to see the full transcription clearly on the page with the video's total duration at the end, and copy it with one click,
**so that** I can use the text immediately without manual reformatting or copy-tedium.

## Acceptance Criteria

> Source: `docs/prd.md` Epic 1, Story 1.5 (verbatim).

1. **AC1:** On successful transcription, the full transcript text is displayed in a scrollable, selectable text area (e.g., `st.text_area` with `height` sized to ~400px or `st.markdown` block — choice left to implementation).
2. **AC2:** The output contains **NO timestamps, NO speaker labels, NO segment markers** — only the raw transcript text as returned by Whisper, with natural paragraphing preserved.
3. **AC3:** Below the transcript text, a final line is rendered: **`Duração total: MM:SS`** where MM is total minutes (no leading zero suppression: 7 minutes → `07`) and SS is remaining seconds (zero-padded). Example: `Duração total: 23:08`.
4. **AC4:** The duration value comes from the `yt-dlp` metadata extracted in Story 1.2 — **NOT recomputed from the audio file**.
5. **AC5:** A **"Copiar"** button is rendered adjacent to the output. Clicking it copies the **full text (transcript + duration line)** to the user's clipboard via `st.clipboard` or a custom JS-injected approach if Streamlit lacks a native clipboard component at the time of implementation.
6. **AC6:** Successful copy displays a transient confirmation message (**"Copiado!"**) that fades or persists for ~2 seconds.
7. **AC7:** After successful rendering, the input field is reset and ready to accept a new URL without page reload.
8. **AC8:** Unit tests cover the MM:SS formatter for: **0s, 30s, 60s, 61s, 599s, 600s (exact 10 min), 3599s (one second under 60 min), 3600s (60 min flat — edge of cap)**.

### Task 1: Create `app/duration.py` — pure MM:SS formatter (AC: 3, 8)

> Architecture §4.1 + §4.3 enumerated `duration.py` as a planned module from day one. This is the **last architectural utility module** to land — after Story 1.5, the §4.1 module set is fully realized.

- [x] Create `app/duration.py` with a single public function:
  ```python
  """MM:SS formatter for video duration display.

  Pure module — no side effects, no external imports beyond stdlib.
  Per architecture §4.3, this is the canonical formatter for the
  `Duração total: MM:SS` line in the rendered output.
  """


  def format_duration(seconds: int) -> str:
      """Format an integer number of seconds as MM:SS with zero-padding.

      Examples:
          0     → "00:00"
          30    → "00:30"
          60    → "01:00"
          61    → "01:01"
          599   → "09:59"
          600   → "10:00"
          3599  → "59:59"
          3600  → "60:00"   (60-min cap edge)

      Args:
          seconds: Non-negative integer number of seconds.

      Returns:
          MM:SS string with zero-padded minutes and seconds.

      Raises:
          ValueError: If `seconds` is negative.
      """
      if seconds < 0:
          raise ValueError(f"seconds must be non-negative, got {seconds}")
      minutes, remainder = divmod(int(seconds), 60)
      return f"{minutes:02d}:{remainder:02d}"
  ```
  - [Source: `docs/architecture.md` §4.1 + §4.3 + Story 1.5 AC3/AC8]
- [x] **AC3 spec compliance:** Zero-padding on BOTH minutes and seconds (`%02d`); no leading zero suppression; format `"MM:SS"`. _All 8 verbatim AC8 cases verify._
- [x] **DO NOT** import streamlit, openai, pydub, or any 3rd-party module. _Verified: only stdlib._
- [x] **DO NOT** use `datetime.time` or `time.strftime`. _Followed: plain `divmod`._

  ```python
  import streamlit.components.v1 as components
  import json


  def _render_copiar_button(text: str) -> None:
      """Render a 'Copiar' button that copies `text` to the user's clipboard.

      Uses minimal JS injection because Streamlit (as of v1.40) lacks a
      native clipboard component. The 'Copiado!' toast persists for ~2 s
      via setTimeout (AC6).

      escaped = json.dumps(text)  # safe JSON-escaping for embedding in JS
      html = f"""
      <button id="copiar-btn"
              style="padding: 8px 16px; font-size: 14px; cursor: pointer;
                     background: #ff4b4b; color: white; border: none;
                     border-radius: 4px;">
          Copiar
      </button>
      <span id="copy-status" style="margin-left: 12px; color: #28a745;
                                     font-weight: 600;"></span>

        const btn = document.getElementById('copiar-btn');
        const statusEl = document.getElementById('copy-status');
        btn.addEventListener('click', async () => {{
          try {{
            await navigator.clipboard.writeText({escaped});
            statusEl.textContent = 'Copiado!';
            setTimeout(() => {{ statusEl.textContent = ''; }}, 2000);
          }} catch (err) {{
            statusEl.textContent = 'Falha ao copiar';
            statusEl.style.color = '#d33';
          }}
        }});
      </script>
      """
      components.html(html, height=60)
  ```

### Streamlit Clipboard Approach (AC5)

Streamlit v1.40 does not have a native `st.clipboard` component. Architecture §3 + §7 don't dictate the clipboard approach; AC5 itself authorizes "a custom JS-injected approach if Streamlit lacks a native clipboard component."

**Decision:** use `streamlit.components.v1.html(...)` with a small inline button + JS that calls `navigator.clipboard.writeText(...)`. This is:

- **Zero new dependencies** — `components.v1` ships with streamlit
- **Browser-native** — no external CDN, no React, no extra JS bundle
- **Safe** — `json.dumps(text)` escapes the transcript before embedding in JS, preventing XSS even on adversarial transcripts
- **Visually matches Streamlit's default red button styling** (`#ff4b4b`)

Alternatives considered and rejected:
- `streamlit-extras` (adds a 3rd-party dependency for a one-line feature)
- `st.code(text)` with built-in copy icon (the icon is small and in the corner; doesn't match the AC's "Copiar button adjacent to output" spec)
- Pure `pyperclip` (server-side, requires X11/clipboard daemon — Railway containers don't have this)

### Empty-Transcript Edge Case

Story 1.4 already handles the "Whisper returned <10 chars" case via `TranscriptionError` (architecture §12 R8). So by the time Story 1.5's rendering code runs, `transcript` is guaranteed to be a non-empty string. We do not need to defensively check for empty/None here.

#### Should-Fix (HIGH priority — but non-blocking)

**Issue: AC7 interpretation is too loose.**

PRD AC7 (verbatim): *"After successful rendering, the input field is reset and ready to accept a new URL without page reload."*

The story currently interprets this as: *"page is interactive; user can edit URL and submit again."*

**This interpretation does NOT honor the spec.** "Reset" in English has a specific meaning: cleared/zeroed. The PRD's word choice is deliberate — if PM (Morgan) had meant "merely usable", he'd have written "the input field remains usable" or "the user can submit a new URL". He wrote "reset", which means clear.

**Recommendation for @dev:** wrap the input + submit button in `st.form(clear_on_submit=True)`. This is the idiomatic Streamlit pattern for "clear input after submit":

```python
with st.form("transcribe_form", clear_on_submit=True):
    url_input = st.text_input(
        label="Cole aqui o link do vídeo",
        key="url_input",
        placeholder="https://...",
    )
    submitted = st.form_submit_button("Transcrever")

if submitted:
    # ... existing pipeline ...
```

**Benefits:**
- Honors AC7 literally (input is cleared after each successful submission)
- Idiomatic Streamlit (no session_state gymnastics)
- 4 lines of code change (wrap existing input/button in `with st.form(...)`, change `st.button` to `st.form_submit_button`)
- The transcript output below is unaffected — it persists across reruns until the next form submit cycle

**Why this matters:** the next time the user types a URL, an empty field is more useful UX than "their old URL still sitting there from 5 minutes ago". The current loose implementation leaves stale state in the input.

**Why non-blocking:** the implementation works either way; users can manually clear the input if they want. But the literal reading is what the PRD specified, and the fix is trivial.
