# PV_PA_036 - Input Resilience vs Vanity Cycle Time

**Type:** Decision Heuristic
**Phase:** 5 (Optimization)
**Agent:** @hybridOps:process-architect

## Purpose

Do not evaluate Workers (Scripts/Automations) using traditional Lean cycle-time metrics. Workers execute fast natively. Determine a Worker's true efficiency strictly by its "Input Resilience": its ability to gracefully handle poorly formatted data, edge-case arrays, and prevent total pipeline breakdown.

## Configuration


## Application

**Process:** Focus dashboard reporting away from "Worker executed in 0.5s". Track how often the Worker halted processing because a CSV had a missing header, throwing an error into the Human lap.

## Decision Tree

```
IF (Executor == Worker) 
THEN PRIMARY KPI = "INPUT RESILIENCE / EXCEPTION HANDLINGS" (NOT PURE SPEED).
```

## Failure Modes

### Vanity Speed
- **Trigger:** Management celebrating that "the routing algorithm processes 10,000 files a minute!" while willfully ignoring the 30% dropout rate generated by unpredicted JSON blocks.

## Validation

**Paradox:** Fast Workers that break on edge cases harm throughput more than slightly slower Workers with robust exception handling.

**Validation Checklist:**
- [ ] Worker exception logs reviewed for unhandled edge cases
- [ ] Fallback rate measured and tracked weekly
- [ ] Input validation coverage assessed (missing headers, malformed JSON, array length edge cases)
- [ ] Speed KPI deprioritized in favor of Resilience KPI
- [ ] Dashboard metrics updated to surface exception handling coverage first
