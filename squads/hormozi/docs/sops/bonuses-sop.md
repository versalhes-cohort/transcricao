## Part 1: Purpose

Transform the perceived value of your offer by strategically stacking bonuses that expand the price-value discrepancy until prospects feel compelled to buy. Bonuses increase the likelihood that people who would otherwise say no will say yes.

**Core Principle:**
```
A single offer is less valuable than the same offer broken into
component parts and stacked as bonuses.

The discrepancy between price and value grows with each bonus until
it becomes "too good to refuse" - breaking the elastic holding the
prospect's wallet in their pocket.
```

**Business Impact:** Bonuses allow you to increase value delivered without cutting price. They put you in a position of strength and goodwill instead of weakness. Each bonus can also become a revenue stream through strategic partnerships.

## Part 2: Scope

### When This SOP Applies
- Creating or enhancing any offer (products, services, programs)
- Closing sales 1-on-1 or through presentations
- Overcoming specific objections to purchase
- Increasing perceived value without lowering price
- Building partner relationships for co-marketing

### When This SOP Does NOT Apply
- Initial offer creation (covered in Trim and Stack - Chapter 10)
- Core product/service delivery design
- Pricing strategy (covered separately)
- Guarantees and risk reversal (covered in separate chapter)

### Business Types
- Any business selling products or services
- Especially effective for high-ticket offers
- Works for B2B and B2C
- Digital and physical products

## Part 4: The Bonus Framework

### Framework Overview

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    BONUS STACKING FRAMEWORK                              │
├──────────────────┬──────────────────┬───────────────────────────────────┤
│ BONUS CREATION   │ BONUS PRESENTATION │ BONUS ENHANCEMENT               │
├──────────────────┼──────────────────┼───────────────────────────────────┤
│ What to offer    │ How to present   │ How to amplify                    │
│ as bonuses       │ them effectively │ their power                       │
├──────────────────┼──────────────────┼───────────────────────────────────┤
│ Tools & lists    │ Name + benefit   │ Add scarcity                      │
│ Recordings       │ Problem relation │ Add urgency                       │
│ Templates        │ Origin story     │ Partner bonuses                   │
│ Scripts          │ Future pacing    │ Commission deals                  │
│ Workshops        │ Price anchor     │ Group discounts                   │
└──────────────────┴──────────────────┴───────────────────────────────────┘
```

### The Value Stacking Principle

From Chapter 14:
```
"Every infomercial continues with 'but wait... there's more!'
They wouldn't use these techniques unless they were effective,
as every second of airtime costs money and must be justified with ROI."
```

The psychological mechanism:
1. Anchor the price to the main offer
2. With each bonus, the price-value discrepancy grows
3. The prospect subconsciously believes: "If these are the bonuses, the main thing must be even MORE valuable"
4. Eventually the discrepancy becomes too large to resist

## Part 5: Process Steps

### PHASE I: BONUS CREATION

#### Step 1.1: Identify Bonus Candidates from Existing Assets
| Field | Value |
|-------|-------|
| **Description** | Inventory all existing assets that could become bonuses |
| **Cognitive Type** | Analysis (asset evaluation) |
| **Automation Potential** | LOW - Requires creative assessment |
| **Executor** | Human |
| **Precondition** | Business has been operating and creating content |
| **Output** | Bonus candidate inventory list |
| **Guardrails** | Must have high perceived value, low delivery cost |
| **Decision Rule** | "High Wow Factor - things you don't want someone to miss" |

**Asset Types to Inventory:**
- Workshops (recorded)
- Webinars (recorded)
- Events (recorded)
- Interviews (recorded)
- Checklists
- Templates
- Scripts
- Tools
- Calculators
- Swipe files

**From Source:** "Create the habit of recording every workshop, every webinar, every event, every interview and use them as additional bonuses."

#### Step 1.2: Create Tools Over Trainings
| Field | Value |
|-------|-------|
| **Description** | Develop tools, checklists, and templates (not more training) |
| **Cognitive Type** | Synthesis + Creation |
| **Automation Potential** | LOW - Requires expertise |
| **Executor** | Human |
| **Precondition** | Problems and solutions identified |
| **Output** | High-value, low-effort bonus assets |
| **Guardrails** | Tools > Training (lower effort perception = higher value) |
| **Decision Rule** | "Invest once, provide infinitely" |

**Value Hierarchy (from Value Equation):**
```
1. Tools & Checklists     → Lowest effort/sacrifice = HIGHEST VALUE
2. Templates & Scripts    → Low effort
3. Done-with-you support  → Medium effort
4. Training/courses       → Higher effort (less valuable as bonus)
```

**From Source:** "Tools and checklists are better than additional trainings (since effort and time are lower with the former, so value is higher. The value equation still reigns supreme)."

#### Step 1.3: Address Specific Obstacles with Each Bonus
| Field | Value |
|-------|-------|
| **Description** | Map each bonus to a specific objection or obstacle |
| **Cognitive Type** | Judgment + Synthesis |
| **Automation Potential** | LOW |
| **Executor** | Human |
| **Precondition** | Customer obstacles documented |
| **Output** | Obstacle-to-bonus mapping |
| **Guardrails** | Each bonus must prove a limiting belief incorrect |
| **Decision Rule** | "Solve the next problem before they encounter it" |

**Bonus Purpose Categories:**
1. Prove their limiting belief is incorrect
2. Solve the next logical problem they'll face
3. Remove a perceived obstacle to success
4. Make success seem more likely

**From Source:** "Each [bonus] should address a specific concern/obstacle in the prospects' minds about why they can't or won't have success (the bonus should prove that their belief is incorrect)."

#### Step 1.4: Determine What Should Be Bonus vs Core Offer
| Field | Value |
|-------|-------|
| **Description** | Decide which deliverables to highlight as bonuses |
| **Cognitive Type** | Judgment |
| **Automation Potential** | LOW |
| **Executor** | Human |
| **Precondition** | Full solution delivery mapped |
| **Output** | Core offer vs bonus categorization |
| **Guardrails** | Bonuses = high-impact but could be overlooked items |
| **Decision Rule** | "Short in duration, high in quality or value" |

**Criteria for Bonus Selection:**
- "Wow Factor" - something you don't want them to miss
- Distinct enough to stand alone
- Short duration but high quality
- Easy to overlook in the main package
- High perceived value but low delivery cost

**From Source:** "Often you have so much 'stuff' you'll be providing customers (good stuff) that valuable nuggets can get lost in the mix. You want to take the most distinct ones that can almost stand alone and highlight them."

| Field | Value |
|-------|-------|

| **Cognitive Type** | Execution |
| **Automation Potential** | HIGH |

**From Source:** "If they say yes, then after they sign up, you let them know about the additional bonuses they'll be getting. This creates a wow experience and reinforces their decision to buy."

#### Step 3.3: If No - Match Bonus to Objection
| Field | Value |
|-------|-------|
| **Description** | Present a bonus that addresses their specific obstacle |
| **Cognitive Type** | Judgment + Negotiation |
| **Automation Potential** | LOW |
| **Executor** | Human |
| **Precondition** | Obstacle identified, matching bonus exists |
| **Output** | Targeted bonus offer |
| **Guardrails** | Bonus must directly address their concern |

**From Source:** "If the person doesn't buy after the first ask, then you present a bonus that matches their perceived obstacle, then ask again."

**Script Framework:**
```
1. Agree with their concern
2. Present the bonus that addresses it
3. Ask: "Fair enough?"
4. Leverage reciprocity - they feel obligated
```

**From Source:** "People have a hard time rejecting reciprocity, so adding a bonus to accommodate, then another, then another, and people will feel almost obligated to buy from you."

### PHASE IV: BONUS ENHANCEMENT

#### Step 4.1: Add Scarcity to Bonuses
| Field | Value |
|-------|-------|
| **Description** | Make certain bonuses available only through this offer |
| **Cognitive Type** | Strategy |
| **Automation Potential** | MEDIUM |
| **Executor** | Hybrid |
| **Precondition** | Exclusive bonuses identified |
| **Output** | Scarcity-enhanced bonus presentation |
| **Guardrails** | Scarcity must be genuine |
| **Decision Rule** | "Available nowhere else" |

**Scarcity Version 1:**
```
"Only people who sign up for program XYZ will have access to
my Bonuses #1, 2, 3 that are never for sale or available
anywhere else other than through this program."
```

**Scarcity Version 2:**
```
"I have 3 remaining tickets for my $5,000 virtual event,
if you buy this program, you can get one of the last 3
tickets as a bonus."
```

**From Source:** "You can further enhance the value of your bonuses by adding scarcity and urgency to the bonuses themselves (which puts this technique on steroids)."

#### Step 4.2: Add Urgency to Bonuses
| Field | Value |
|-------|-------|
| **Description** | Make certain bonuses available only for immediate action |
| **Cognitive Type** | Strategy |
| **Automation Potential** | MEDIUM |
| **Executor** | Hybrid |
| **Precondition** | Time-limited bonuses identified |
| **Output** | Urgency-enhanced bonus presentation |
| **Guardrails** | Urgency must be genuine |
| **Decision Rule** | "Reward the action-takers" |

**Urgency Script:**
```
"If you buy today, I will add the XYZ bonus which normally
costs $1,000, for free. And I'll do this because I want to
reward the action-takers."
```

**Key Distinction:**
- Scarcity = Not time-limited but only available through this program
- Urgency = Time-limited, buy TODAY or lose the bonus

**From Source:** "The first two examples are not time limited. They state that if you buy the program, you get things you normally wouldn't be able to get. The urgency bonus is about them buying today, and if they don't buy today, they lose out on those bonuses."

## Part 6: Decision Rules

### DR-001: Bonus vs Core Offer Classification
```yaml
rule: BONUS_CLASSIFICATION
trigger: Deciding what to highlight as bonus
condition:
  - IF high_wow_factor AND short_duration THEN "BONUS"
  - IF could_stand_alone AND easily_overlooked THEN "BONUS"
  - IF critical_to_main_outcome THEN "CORE OFFER"
action: Highlight distinct, high-impact items as bonuses
confidence: HIGH
source: "Short in duration but high in quality or value"
```

### DR-002: Tool vs Training Priority
```yaml
rule: BONUS_TYPE_PRIORITY
trigger: Creating new bonus assets
condition:
  - IF can_be_tool_or_checklist THEN "HIGHEST PRIORITY"
  - IF can_be_template_or_script THEN "HIGH PRIORITY"
  - IF must_be_training THEN "LOWER PRIORITY"
action: Convert trainings into tools when possible
confidence: HIGH
source: "Tools and checklists are better than additional trainings"
```

### DR-003: Discount Request Response
```yaml
rule: NO_DISCOUNT_POLICY
trigger: Prospect asks for lower price
condition:
  - IF discount_requested THEN add_bonus() NOT reduce_price()
  - IF still_objecting THEN add_another_bonus()
  - NEVER reduce_main_offer_price
action: Stack bonuses until value overwhelms objection
confidence: HIGH
source: "Adding bonuses to increase value is far superior to cutting prices"
```

### DR-004: 1-on-1 Bonus Timing
```yaml
rule: ONE_ON_ONE_BONUS_TIMING
trigger: 1-on-1 sales conversation
condition:
  - IF first_close_attempt THEN present_offer_without_bonuses()
  - IF they_say_yes THEN reveal_bonuses_as_surprise()
  - IF they_say_no THEN present_obstacle_matching_bonus()
action: Ask for sale first, use bonuses strategically
confidence: HIGH
source: "When selling one-on-one, you ask for the sale first, before offering the bonuses"
```

### DR-005: Scarcity vs Urgency Selection
```yaml
rule: SCARCITY_URGENCY_SELECTION
trigger: Enhancing bonus appeal
condition:
  - IF bonus_exclusively_available_through_this_offer THEN use_scarcity()
  - IF bonus_time_limited THEN use_urgency()
  - IF both_applicable THEN combine_both()
action: Match enhancement type to bonus characteristics
confidence: HIGH
source: "The first two examples are not time limited... The urgency bonus is about them buying today"
```

### DR-006: Partner Bonus Evaluation
```yaml
rule: PARTNER_BONUS_VALUE
trigger: Evaluating partner bonus opportunity
condition:
  - IF partner_not_competing AND serves_same_customer THEN "GOOD FIT"
  - IF can_negotiate_commission THEN "HIGH VALUE"
  - IF solves_next_logical_problem THEN "EXCELLENT FIT"
action: Pursue partnerships that create triple-win
confidence: HIGH
source: "What is the next natural thing they will want?"
```

## Part 7: Exceptions & Edge Cases

### Exception 1: No Existing Bonus Assets
**Trigger:** New business with no recordings or tools
**Action:** Prioritize creating high-value, low-cost assets
**Priority Creation Order:**
1. Checklists (fastest to create)
2. Templates and scripts
3. Tools and calculators
4. Record next workshop/webinar

### Exception 2: Bonuses Overwhelming Core Offer
**Trigger:** So many bonuses that main offer gets lost
**Response:** Restructure presentation
**Solution:**
- Lead with core offer value
- Stack 3-5 key bonuses
- Save additional bonuses for post-purchase surprise

### Exception 3: Partner Refuses Commission
**Trigger:** Partner wants exposure but won't pay referrals
**Response:** Accept if bonus value still high
**Condition:** Partner bonus must still make your offer more valuable

### Exception 4: Bonus Value Exceeds Main Offer
**Trigger:** Bonus total appears more valuable than core
**Response:** This is the GOAL, not a problem
**From Source:** "The value of the bonuses should eclipse the value of the core offer"

### Exception 5: Customer Only Wants Bonus
**Trigger:** Prospect wants to buy just the bonus
**Response Options:**
1. Explain bonus only available through main offer (scarcity)
2. Consider creating separate product from bonus
3. Maintain integrity of offer structure

## Appendix C: Bonus Type Decision Tree

```
START: What problem does this solve?

├── Is it a tool, template, or checklist?
│   └── YES → HIGH VALUE BONUS (lowest effort for customer)
│   └── NO ↓
│
├── Is it a recording of something valuable?
│   └── YES → GOOD BONUS (workshop, webinar, interview)
│   └── NO ↓
│
├── Is it additional training?
│   └── YES → Convert to tool/checklist if possible
│   │   └── If not possible → LOWER VALUE BONUS
│   └── NO ↓
│
├── Is it access to a community or support?
│   └── YES → MEDIUM VALUE BONUS
│   └── NO ↓
│
└── Is it a partner service/product?
    └── YES → HIGH VALUE if solves next logical problem
    └── NO → Reconsider if truly bonus-worthy
```
